1. الفلسفة العامّة لاختيار التقنيّات

عند بناء موقع لمنظّمة غير ربحيّة بميزانيّة صفر، نتّبع 4 مبادئ:

  1. المجانيّة الكاملة في البداية: كل خدمة تستخدم الطبقة المجانيّة (Free Tier) لها
  2. القابليّة للتوسّع: لو احتجت يوماً للترقية، يكفي دفع المال — لا حاجة لإعادة بناء
  3. النضج والاستقرار: نختار خدمات تأسّست منذ سنوات ولها سمعة قويّة
  4. التخصّص في كلّ مكوّن: لا نستخدم خدمة واحدة لكلّ شيء — كلّ خدمة مُتخصّصة في عمل واحد تتقنه
💡 المبدأ الأهمّ: في عصرنا الحاضر يمكن لشخص واحد بناء موقع ضخم بربط 4-5 خدمات مجانيّة (المعروف بـ Jamstack). لا حاجة لاستئجار خوادم.

المكوّنات الأساسيّة لأيّ موقع حديث

المكوّنالمهمّةأمثلة على الخدمات
الواجهة الأماميّةما يراه المستخدم في المتصفّحReact, Vue, Angular, Svelte
الاستضافةتوصيل الموقع للعالم بسرعةVercel, Netlify, Cloudflare Pages
قاعدة البياناتتخزين البيانات بشكل دائمSupabase, Firebase, PlanetScale
المصادقةتسجيل الدخول والصلاحيّاتSupabase Auth, Clerk, Auth0
تخزين الملفّاتالصور والمرفقاتSupabase Storage, S3, Cloudinary
البريدإرسال إيميلاتResend, SendGrid, Postmark
المراقبةتأكّد أنّ الموقع يعملUptimeRobot, Pingdom, Better Uptime

2. الواجهة الأماميّة (Frontend)

ما اخترنا: React + Vite + Tailwind CSS

React

مكتبة من Facebook (Meta) لبناء واجهات تفاعليّة. تأسّست 2013، الأكثر استخداماً عالمياً.

الإيجابيّاتالسلبيّات
• مجتمع ضخم — كلّ مشكلة تجد لها حلّاً
• مكتبات لكلّ شيء (تقاويم، رسوم بيانيّة، جداول)
• سوق العمل ضخم
• مرن — يناسب كلّ أنواع المشاريع 		• منحنى تعلّم متوسّط
• يحتاج بناء (لا يعمل بـHTML خام)
• قرارات كثيرة (أيّ توجيه، أيّ مدير حالة...)

البدائل وميّزاتها

التقنيّةمتى تختارهامتى تتجنّبهاصعوبة التعلّم

HTMX + جانب الخادمللمواقع البسيطة جدّاًللتطبيقات المعقّدةمنخفضة

Vite

أداة بناء سريعة جدّاً (من مؤسّس Vue). تجمع وتُحسّن الكود قبل الإنتاج.

البديل المعروف: Create React App (مهجور الآن من Facebook) أو Webpack (أبطأ بكثير). Vite هو المعيار الجديد.

Tailwind CSS

إطار CSS بفلسفة "Utility-First" — تستخدم classes صغيرة بدل كتابة CSS مخصّص.

<!-- بدل كتابة class مخصّص: -->
<div class="my-card">...</div>

<!-- في Tailwind: -->
<div class="bg-white rounded-xl border p-4 shadow-md">...</div>
الإيجابيّاتالسلبيّات
• ما تحتاج تكتب CSS غالباً
• الحجم النهائي صغير جدّاً (يحذف غير المستخدم)
• تصميم متجاوب جاهز 		• HTML يصبح طويلاً
• منحنى تعلّم للأسماء (px-4 = padding-x = 16px)

البدائل

  • Bootstrap: قديم، مكوّنات جاهزة جدّاً، أكثر عاميّة
  • Material UI / Chakra UI: مكوّنات React جاهزة (سريع للبدء، أصعب للتخصيص)
  • CSS الخام: أكثر تحكّماً، أكثر عملاً

3. الخادم وقاعدة البيانات (Backend)

ما اخترنا: Supabase

Supabase هو "Backend-as-a-Service" — يقدّم لك قاعدة بيانات + مصادقة + تخزين ملفّات + REST API + Realtime — كلّ هذا بدون كتابة سطر backend واحد.

تأسّست 2020. مفتوحة المصدر. مبنيّة على PostgreSQL (الأقوى عالمياً في قواعد البيانات).

الطبقة المجانيّة (Free Tier) — ما الذي نحصل عليه

  • قاعدة بيانات PostgreSQL 500 MB
  • تخزين ملفّات 1 GB
  • مصادقة 50,000 مستخدم نشط شهرياً
  • 2 مشروع نشط (لا يُستخدمان لمدّة 7 أيّام = يُجمَّد)
  • 500 MB نقل بيانات شهرياً
⚠ التجميد بعد 7 أيّام عدم استخدام: نحلّ هذا بـ Cron-job.org يُرسل ping يوميّاً لإبقاء المشروع نشطاً.

البدائل وميّزاتها

الخدمةالإيجابيّاتالسلبيّاتالمجانيّة
Supabase ⭐ PostgreSQL كامل، مفتوح المصدر، يمكن استضافته بنفسك الجيل الأحدث، أقلّ نضجاً من Firebase مجاني
Firebase من Google، النضج، مكتبات لكلّ المنصّات قاعدة بيانات NoSQL غريبة، مغلق المصدر، صعب التحميل طبقة محدودة
PocketBase ملفّ تنفيذي واحد، تستضيفه بنفسك أصغر، أقلّ ميزات، مكتوب بـGo مجاني تماماً
PlanetScale MySQL ممتاز للتوسع الكبير أنهى الطبقة المجانيّة 2024 $29+/شهرياً
Neon PostgreSQL سحابي، توسّع تلقائي أحدث، أقلّ ميزات طبقة محدودة

ميزات Supabase التي استخدمناها

  • PostgreSQL Tables: 11 جدولاً للمستودع والمساحات والصناديق والأغراض
  • RLS (Row Level Security): تحكّم بمن يقرأ ماذا — مثل أنّ المؤسّس فقط يرى مفتاح Resend
  • RPC Functions: 16 دالّة SQL مخصّصة (مثل `move_box_to_position`)
  • Triggers: محفّزات تلقائيّة (مثل بريد الترحيب)
  • pg_net Extension: استدعاء HTTP من قاعدة البيانات
  • Storage Buckets: لتخزين صور الصناديق والأغراض
  • Auth: تسجيل الدخول بالبريد + الإيميلات التلقائيّة

4. الاستضافة (Hosting)

ما اخترنا: Vercel

Vercel هي شركة مؤسّس Next.js. تستضيف المواقع الأماميّة بسرعة عالميّة عبر CDN.

الطبقة المجانيّة

  • 100 GB نقل بيانات شهرياً
  • مشاريع غير محدودة
  • SSL تلقائي (HTTPS)
  • نطاق فرعي مجاني (`*.vercel.app`)
  • نشر تلقائي عند كلّ git push
  • Preview deployments لكلّ pull request

البدائل

الخدمةالإيجابيّاتالسلبيّاتالحدّ المجاني
Vercel ⭐ الأسرع، أحسن DX (تجربة المطوّر) محدود نقل البيانات 100 GB/شهر
Netlify الأقدم، نفس الفكرة، Forms مجاني أبطأ قليلاً من Vercel 100 GB/شهر
Cloudflare Pages أسرع CDN عالمياً، نقل بيانات لا محدود أقلّ ميزات تلقائيّة غير محدود
GitHub Pages دمج مباشر مع GitHub للمواقع الثابتة فقط، لا توجد بيئات معاينة 1 GB التخزين
💡 لو احتجت توسّع: انتقل إلى Cloudflare Pages — مجانيّ بالكامل بلا حدود نقل، ويمتلك أكبر شبكة CDN في العالم.

5. المصادقة (Authentication)

ما اخترنا: Supabase Auth (مدمَج)

بدل استخدام خدمة منفصلة، استخدمنا المصادقة المدمجة في Supabase. هذا يبسّط الأمور.

ما تقدّمه

  • تسجيل دخول بـEmail/Password
  • تأكيد البريد تلقائياً (يمكن تعطيله للتطوير)
  • إعادة تعيين كلمة السرّ
  • JWT tokens للأمان
  • OAuth (Google, GitHub, ...)— لم نستخدمه
  • Magic Links (دخول بدون كلمة سرّ) — لم نستخدمه

البدائل

الخدمةالميزة الأبرزالسلبيّةالمجانيّة
Supabase Auth ⭐مدمَجة مع قاعدة البياناتأبسط من Auth050K مستخدم
Clerkأحلى UI جاهزة، إدارة سهلةمنفصلة عن قاعدة البيانات10K مستخدم
Auth0الأقدم، الأكثر ميزاتغالٍ بعد المجاني، معقّد7K مستخدم
Firebase Authسهلة جدّاًمغلقة، ربط مع Google50K مستخدم

6. تخزين الملفّات (File Storage)

ما اخترنا: Supabase Storage

للصور (صور الأغراض والصناديق). مدمَج مع Supabase = سياسات أمن مشتركة.

كيف يعمل عندنا

  1. المستخدم يختار صورة من جهازه
  2. المتصفّح يضغطها (browser-image-compression) إلى ~500 KB
  3. تُرفَع لـ Supabase Storage في bucket اسمه `sraj-photos`
  4. يُحفَظ الرابط في حقل `photo_url` للغرض/الصندوق

البدائل

الخدمةالتخصّصالمجانيالميزة الفريدة
Supabase Storageعام1 GBRLS مدمج
Cloudinaryالصور والفيديو25 GB + 25 GB نقلتحويل وضغط تلقائيّ
AWS S3عام5 GB سنة واحدةالأقوى للتوسّع
Cloudflare R2عام، متوافق مع S310 GBبدون رسوم نقل!
Backblaze B2تخزين بارد10 GBالأرخص للتوسّع
للصور تحديداً: Cloudinary ممتازة لأنّها تحوّل الصور تلقائياً (resize, webp, إلخ) عند الطلب. لكن تعقّدت Free Tier فيها مؤخّراً.

7. البريد الإلكتروني (Email Services)

ما اخترنا: Resend

Resend خدمة بريد حديثة (2022)، صنعها مؤسّسو React Email. بسيطة جدّاً.

استخداماتنا

  1. تنبيهات تأخّر الإرجاع اليوميّة (GitHub Actions cron)
  2. بريد الترحيب التلقائي (DB trigger via pg_net)

الطبقة المجانيّة

  • 3,000 رسالة/شهر (100/يوم)
  • 1 نطاق فقط (نطاقهم: `resend.dev`)
  • Sandbox mode — ترسل لأيّ بريد بدون تحقّق
⚠ التحقّق من النطاق (Domain Verification): لإرسال بريد من نطاقك (مثل noreply@sraj.org.sa) تحتاج تحقّق DNS — ميزة مجانيّة لكن تحتاج صلاحيّات على نطاق الجمعيّة.

البدائل

الخدمةالمجاني/شهرالإيجابيّاتالسلبيّات
Resend ⭐3,000الأسهل، API نظيفةجديدة
SendGrid100/يومالأقدم، الأشهرUI معقّد، محدود
Mailgun100/يوم 3 أشهر فقطقوي للمطوّرينالمجاني محدود زمنياً
Postmark100/شهرالأسرع تسليماًمحدود جدّاً مجاناً

8. المراقبة والاستقرار (Monitoring)

ما اخترنا: ثلاث طبقات

1. UptimeRobot — مراقبة وقت التشغيل

يفحص الموقع كلّ 5 دقائق ويرسل بريد إذا توقّف.

  • مجاني: 50 monitor، فحص كلّ 5 دقائق
  • تنبيه فوريّ على الإيميل عند التوقّف وعند العودة
  • صفحة حالة عامّة (status page)

2. Cron-job.org — إبقاء قاعدة البيانات نشطة

Supabase Free Tier يُجمِّد المشروع بعد 7 أيّام بلا استخدام. حلّنا: ping يومي.

  • مجاني: 50 cron job
  • دقّة الجدولة دقيقة واحدة
  • سجلّ كامل للنتائج

3. GitHub Actions — نسخ احتياطيّة + تنبيهات

GitHub يُتيح 2,000 دقيقة CI/CD مجانيّة شهرياً للمستودعات الخاصّة. عندنا workflow يوميّ ينسخ قاعدة البيانات JSON إلى فرع `backups`.

البدائل للمراقبة

الخدمةالتخصّصالمجاني
UptimeRobot ⭐وقت تشغيل50 monitor
Better Uptimeوقت تشغيل + status page جميلة10 monitors
Pingdomالأقدم، مدفوع غالباً1 monitor
Sentryتتبّع الأخطاء (error tracking)5K حدث/شهر
LogRocketتسجيل جلسات المستخدمين1K جلسة/شهر

9. التحكّم في الإصدارات و CI/CD

ما اخترنا: Git + GitHub + GitHub Actions

Git

أداة التحكّم في الإصدارات الأقدم والأقوى. بدون بديل عملي.

GitHub

منصّة استضافة Git (مملوكة لـMicrosoft منذ 2018).

البدائل

  • GitLab: مفتوح المصدر، يمكن استضافته بنفسك، CI/CD مدمج أقوى
  • Bitbucket: من Atlassian، تكامل مع Jira
  • Gitea: استضافة ذاتيّة خفيفة

GitHub Actions

أتمتة المهامّ — تشغيل سكربتات عند أحداث (push, schedule, manual).

عندنا: 4 workflows (backup, notify-overdue, create-stairway, fix-stairway)

البدائل لـCI/CD

  • GitLab CI: الأقوى
  • CircleCI: مستقل، مجاني محدود
  • Jenkins: استضافة ذاتيّة، تقليديّ

التوصية النهائيّة

للاستخدام الحاليّ (المؤسّس + 10 متطوّعين كحدّ أقصى):
  • الموقع مستقرّ جدّاً
  • التكلفة صفر دولار
  • المخاطر التي يجب الانتباه لها: تشديد RLS قبل دعوة متطوّعين
عند التوسّع (مستودعات أخرى، 50+ مستخدم):
  • ادفع $25/شهر لـ Supabase Pro (8GB قاعدة بيانات + 100GB تخزين + اختيار المنطقة)
  • اشدّد RLS بناءً على عضويّة المستودع
  • أضِف Sentry لتتبّع الأخطاء
  • اختبارات أساسيّة (لا داعي لتغطية كاملة)

12. المعرفة المستفادة لمشاريع قادمة

قواعد ذهبيّة تعلّمناها

  1. ابدأ بالبيانات أوّلاً: قبل كتابة أيّ كود واجهة، اقرأ كيف ستُخزَّن بياناتك. التخطيط في قاعدة البيانات يوفّر أسابيع لاحقاً.
  2. Soft Delete دائماً: لا تحذف من قاعدة البيانات نهائياً أبداً. حذف خاطئ يساوي كارثة. ضع `deleted_at` بدلاً من ذلك.
  3. Migrations مرقّمة: كلّ تغيير في قاعدة البيانات يكون ملفّاً منفصلاً مرقّماً. لا تعدّل ملفّاً قديماً — أنشئ جديداً.
  4. Vercel + Supabase + GitHub = 90% من احتياجاتك: لا تُعقّد الأمور.
  5. RLS من البداية: لا تترك سياسات الأمان للنهاية. ابدأها ضيّقة وفكّها عند الحاجة، لا العكس.
  6. اكتب نسخ احتياطيّة قبل الإطلاق: قبل أيّ مستخدم حقيقيّ، تأكّد أنّك تستطيع استرجاع كلّ شيء.
  7. UptimeRobot قبل الإطلاق: تنبيه فوري لو تعطّل شيء قبل أن يكتشفه المستخدم.
  8. Tailwind CSS = أوفر وقت كبير: لا تعد لكتابة CSS بعد تجربتها.
  9. RTL ليس مشكلة: dir="rtl" في HTML + flex-direction: row يكفيان لمعظم التصاميم.
  10. المؤسّس هو نفسه DBA: لو لم تُعدّ الأمور لتكون قابلة للإدارة من واجهة، ستندم.

الخطأالدرس
استخدام `ORDER BY code` لترتيب الصناديق النصوص تُرتَّب أبجدياً (`A-1-10` قبل `A-1-2`). استخدم أعمدة رقميّة.
القيد UNIQUE الصارم على رمز الصندوق منع إعادة استخدام الرموز بعد الحذف. الحلّ: فهرس فريد جزئي `WHERE deleted_at IS NULL`.
محاولة `ALTER DATABASE` في Supabase SQL Editor SQL Editor لا يعمل بصلاحيّات superuser. استخدم جدول `app_config` بدلاً منه.
سياسة RLS متساهلة (`auth all`) كلّ مُسجَّل يستطيع تعديل أيّ شيء. لاحقاً نحتاج تشديد.
الاعتماد على `target` نصّياً في activity_log عند إعادة تسمية صندوق، يفقد السجل ربطه. الحلّ: `target_id` (UUID).
